# Краткий гайд по Git

Данный документ представляет собой справочное руководство по основным командам системы контроля версий Git. Он предназначен для быстрого ознакомления и последующего использования в повседневной разработке.

## Инициализация и конфигурация
```bash
git init
```
**Tdt**

    git init
0
Инициализирует новый локальный репозиторий в текущей директории.

    git clone <url>
Клонирует удалённый репозиторий по указанному URL в новую локальную директорию.

    git config --global user.name "Имя"
Устанавливает глобальное имя пользователя, которое будет ассоциироваться с вашими коммитами.

    git config --global user.email "email@example.com"
Устанавливает глобальный email, используемый для идентификации автора коммитов.

## Основные операции с изменениями

    git status
Показывает текущее состояние рабочей директории и индекса (staging area): какие файлы изменены, добавлены или не отслеживаются.

    git add <файл>
Добавляет указанный файл в индекс (staging area) для последующего включения в коммит.

    git add .
Добавляет все изменения в рабочей директории (новые, изменённые и удалённые файлы) в индекс.

    git restore <файл>
Отменяет локальные изменения в указанном файле, возвращая его к состоянию последнего коммита.

    git restore --staged <файл>
Убирает файл из индекса (staging area), не затрагивая его содержимое в рабочей директории.

## Коммиты

    git commit -m "Сообщение"
Фиксирует изменения из индекса в истории репозитория с сопроводительным сообщением.

    git commit --amend
Позволяет изменить последний коммит: обновить его сообщение, добавить или убрать изменения.

    git log
Отображает полную историю коммитов в обратном хронологическом порядке с подробной информацией (хеш, автор, дата, сообщение).

    git log --oneline
Представляет историю коммитов в компактном виде — по одному коммиту на строку с сокращённым хешем и сообщением.

## Ветвление и слияние

    git branch
Выводит список всех локальных веток; текущая ветка помечена звёздочкой.

    git branch <имя>
Создаёт новую ветку с указанным именем, не переключаясь на неё.

    git checkout <ветка>
Переключает рабочую директорию на указанную ветку. Также может использоваться для восстановления файлов (устаревшее для этой цели).

    git switch <ветка>
Современная альтернатива `git checkout` исключительно для переключения между ветками.

    git merge <ветка>
Выполняет слияние указанной ветки в текущую активную ветку.

    git branch -d <ветка>
Удаляет указанную ветку, если она уже была полностью слита в текущую ветку. Для принудительного удаления используйте `-D`.

## Работа с удалённым репозиторием

    git remote -v
Показывает список настроенных удалённых репозиториев и соответствующие URL-адреса (с указанием операций fetch/push).

    git remote add origin <url>
Добавляет новый удалённый репозиторий с именем `origin`, обычно указывающий на основной репозиторий на GitHub/GitLab и т.п.

    git push -u origin <ветка>
Отправляет локальную ветку в удалённый репозиторий и устанавливает связь (upstream), позволяя в дальнейшем использовать `git push` и `git pull` без указания параметров.

    git pull
Выполняет извлечение (`fetch`) изменений из соответствующей удалённой ветки и автоматически сливает их в текущую ветку (`merge`).

    git fetch
Загружает все изменения из удалённого репозитория, не изменяя рабочую директорию и не сливая ветки. Позволяет безопасно просматривать обновления.

## Отмена и восстановление

    git revert <хеш>
Создаёт новый коммит, который отменяет изменения, внесённые указанным коммитом. Безопасен для совместной работы, так как не переписывает историю.

    git reset --soft <хеш>
Перемещает указатель текущей ветки на указанный коммит, сохраняя все последующие изменения в индексе. Полезно для объединения коммитов.

    git reset --hard <хеш>
Полностью откатывает историю, индекс и рабочую директорию до состояния указанного коммита. Все последующие изменения будут безвозвратно утеряны — используйте с осторожностью.

> **Важно:** `git reset --hard` переписывает историю и не должен применяться к веткам, общим для нескольких разработчиков. Для безопасной отмены изменений в общих ветках предпочтителен `git revert`.
